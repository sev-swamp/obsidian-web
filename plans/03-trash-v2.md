# План 3. Корзина 2.0: надёжное восстановление и настоящее удаление

Статус: этапы 1–4 реализованы (2026-07, ADR-0005); этап 5
(переписывание истории) отложен
Приоритет: высокий — текущая корзина молча теряет файлы из вида и содержит
дыры в ACL; «очистка» не удаляет данные.
Зависимости: планы 1–2 (реализованы). Этапы 1–4 независимы от этапа 5
(переписывание истории) — этап 5 можно отложить.

## 1. Цель и мотивация

Корзина сейчас — производное от git-лога: список удалённых файлов каждый
раз вычисляется сканированием коммитов (`packages/history/git.go` →
`Deleted`). Анализ выявил проблемы, по убыванию критичности:

1. **Горизонт корзины ~2000 коммитов** (`maxWalk = 2000`). Каждое
   сохранение — коммит, поэтому на активном vault файл, удалённый пару
   недель назад, молча исчезает из корзины: контент остаётся в git, но
   восстановить через UI нельзя. Список корзины и restore — O(проход по
   всему логу).
2. **Purge скрывает путь навсегда.** `.git/obsidianweb-trash-purged.json`
   хранит «голые» пути: если файл очистили из корзины, создали новый с
   тем же именем и удалили — новое удаление скрыто навсегда.
3. **«Очистить всё» чистит не всё**: `handleTrashPurgeAll` берёт
   `Trash(0)`, а `Deleted(0)` подставляет лимит 100 — очищаются только
   последние 100 записей.
4. **ACL не проверяется в purge**: `handleTrash` фильтрует список через
   `allowRead`, но `handleTrashPurge`/`handleTrashPurgeAll` — нет.
   Пользователь с `trash:purge`, ограниченный по папкам, может скрывать
   чужие записи корзины по любому пути.
5. **Purge не удаляет данные**: контент навсегда остаётся в git-истории и
   доступен любому с `history:read`. Для чувствительных заметок нужно
   настоящее безвозвратное удаление.
6. **Restore не сохраняет источник**: коммит восстановления — просто
   `restore: <path>`, из какой ревизии восстановили — неизвестно.
7. **Кнопка восстановления на верхней ревизии «не работает»**: панель
   показывает diff «что ревизия изменила», а restore восстанавливает
   состояние *на момент* ревизии. Для верхней записи это текущий файл:
   сервер отвечает 200, но ничего не меняется — выглядит как немой отказ.
8. Мелочи: restore перезаписывает текущий файл без optimistic lock;
   в режиме `history.mode: external` удаление безвозвратно сразу, а UI
   не предупреждает.

## 3. Дизайн

### 3.1. Явный индекс корзины (уход от log-scan)

Файл `.git/obsidianweb-trash.json` (не коммитится, тот же паттерн, что
purged-список):

```json
[
  {"path": "Projects/Old.md", "restoreRev": "<hash родителя delete-коммита>",
   "actor": "igor", "time": "2026-07-18T12:00:00Z", "deleteRev": "<hash delete-коммита>"}
]
```

- Запись добавляется в `Record(..., "delete")` сразу после коммита
  (там же известен hash и родитель).
- Запись удаляется при: restore этого пути, создании файла по этому пути
  (`Record` с action `create`/`save`/`external` по пути из индекса), purge.
- `Deleted(limit)` читает индекс (O(1), сортировка по времени, лимит) —
  без прохода по логу и без горизонта 2000 коммитов.
- **Миграция**: при первом старте с пустым индексом выполнить один
  legacy log-scan (текущий код `Deleted`) и наполнить индекс; после этого
  scan-код удаляется либо остаётся только в миграции.
- Запись файла — под существующим `g.mu`, атомарно (tmp + rename), как
  и purged-список.

### 3.2. Честный purge (фиксы 2–4)

- Purged-запись — пара `{path, deleteRev}`, а не голый путь: скрывается
  конкретное удаление, новый файл с тем же именем живёт своей жизнью.
  С индексом из §3.1 purged-файл вообще вырождается: purge просто удаляет
  запись из индекса (контент при этом остаётся в git — см. §3.4);
  отдельный purged-список остаётся только для этапа 5 (что уже
  «приговорено» к вычистке из истории).
- `handleTrashPurge`: проверка `aclAccess(path) >= AccessWrite` (симметрия
  с restore).
- `handleTrashPurgeAll`: брать полный список (после §3.1 «полный» дёшев),
  фильтровать через `allowRead` — очищается только то, что пользователь
  видит в своей корзине.
- В API-ответе и UI переименовать смысл: «Убрать из корзины» (контент
  остаётся в истории), чтобы не обещать удаление данных.

### 3.3. Метаданные восстановления (фикс 6)

- Расширить порт: `Record(actor, path, action, detail string)` (detail —
  пустая строка для обычных действий; реализаций две: git и тестовый фейк).
- Для restore: первая строка коммита не меняется (`restore: <path>` —
  парсер action в `revisionFrom` не трогаем), в тело добавляется трейлер
  `Restored-From: <rev>`.
- `core.Revision` получает поле `SourceRev string` (json `sourceRev`),
  парсится из тела коммита в `revisionFrom`.
- `RestoreDeleted` получает то же бесплатно (идёт через `RestoreNote`).
- UI: в `HistoryPanel` у restore-записей бейдж «← abc1234»; клик
  подсвечивает/прокручивает к ревизии-источнику (если она в списке).

### 3.4. Настоящее удаление: переписывание истории (фикс 5)

Отдельная админ-операция «Удалить безвозвратно» (право `trash:destroy`,
по умолчанию только admin):

- **Replay всей истории на go-git**: обход коммитов от корня к HEAD,
  из каждого tree исключается целевой путь, коммиты, ставшие пустыми,
  пропускаются; строится карта `старый hash → новый hash` (родители
  перемаппливаются). Ветка переключается на новую голову одним движением
  в конце — прерывание ничего не ломает (новые объекты аддитивны).
- После переключения — `Repository.Prune` + `RepackObjects`, иначе блобы
  физически остаются в `.git` и «безвозвратность» фиктивна. Это
  обязательная часть операции, не оптимизация.
- Глобальный лок истории на время операции (все `Record/Log/...` ждут);
  выполняется в фоне, прогресс — в лог и в ответ API.
- Все сохранённые ревизии перемапливаются по карте хэшей: индекс корзины
  (§3.1), purged-список. `restoreRev` вычищаемого пути удаляется вместе
  с записью.
- Работает только в `history.mode: managed`; в `external` — 409 с
  объяснением (чужой репозиторий не переписываем).
- API: `POST /api/trash/destroy {path}` (+ подтверждение в UI с явным
  «данные будут удалены из истории безвозвратно»).

### 3.5. UX восстановления в истории (фиксы 7–8)

- У верхней ревизии кнопка меняется на **«Откатить это изменение»** —
  restore на родителя этой ревизии (то действие, которое пользователь
  ожидает, глядя на diff). У остальных остаётся «Восстановить эту
  версию».
- Бэкенд: если restore не изменил файл (контент совпал), ответ
  `{status: "unchanged"}`; фронтенд показывает тост «содержимое уже
  совпадает» — поведение никогда не выглядит немым отказом.
- `RestoreNote` дополняется сравнением: если на диске контент новее
  последней ревизии (несохранённые в историю правки — внешние
  редакторы), перед перезаписью зафиксировать текущее состояние
  (`Record(..., "save")`) — правка не теряется даже при откате.
- В режиме `external` UI удаления предупреждает: «история выключена,
  восстановление будет невозможно» (флаг режима уже доступен через
  настройки/статус API — при отсутствии добавить в `GET /api/config`).

самый тяжёлый и наименее срочный.

## 5. Риски и решения

- **Рассинхронизация индекса с репозиторием** (правки git руками, сбой
  между коммитом и записью индекса): при старте — быстрая валидация
  записей (файл всё ещё отсутствует? ревизия существует?), битые записи
  выбрасываются; команда пересборки индекса из лога (миграция §3.1
  переиспользуется).
- **Replay на большом vault**: O(коммиты × дерево). Замерить на 10k
  коммитов; выполняется в фоне под глобальным локом, UI показывает
  «история недоступна, идёт очистка». При неприемлемой длительности —
  оптимизация: переписывать только коммиты, затрагивающие путь, с
  переносом остальных через кэш деревьев.
- **Prune/Repack в go-git** менее обкатан, чем в CLI git: интеграционный
  тест «после destroy блоб недостижим и физически отсутствует в .git»
  (поиск hash в objects/pack) — обязателен.
- **Прерывание replay**: новые объекты аддитивны, ref переключается в
  конце — повторный запуск идемпотентен; недописанные объекты вычистит
  следующий prune.
- **Совместимость API**: `/api/trash*` контракты не меняются (только
  дополняются `deleteRev`/`sourceRev`) — фронтенд обновляется в тех же
  этапах.

## 6. Критерии приёмки

1. Файл, удалённый 5000 коммитов назад, виден в корзине и
   восстанавливается (тест с генерацией истории).
2. Сценарий «удалили → очистили из корзины → создали одноимённый →
   удалили» — второй файл виден в корзине.
3. «Очистить всё» при 150+ записях очищает все; пользователь с ACL видит
   и очищает только свои пути; purge чужого пути → 403.
4. Restore-запись в истории показывает ревизию-источник; клик ведёт к ней.
5. Верхняя ревизия: «Откатить это изменение» возвращает файл к родителю;
   restore без изменений отвечает `unchanged`, UI показывает тост.
6. После «Удалить безвозвратно» контент недоступен ни через
   `/api/history`, ни через diff, ни физически в `.git` (grep по
   objects); остальная история файлов не изменилась по содержимому.
7. `go test ./...` зелёный; новые тесты: индекс (add/remove/миграция/
   валидация), purge-фиксы, Restored-From парсинг, replay (карта хэшей,
   пустые коммиты, prune).
8. Документация: docs/api.md (destroy, unchanged, sourceRev), справка в
   UI (корзина: «убрать из корзины» vs «удалить безвозвратно»),
   docs/adr — ADR о явном индексе корзины и переписывании истории.
