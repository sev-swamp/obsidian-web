# План 1. Фундамент: безопасная совместная работа с файлами

Статус: готово (реализован полностью)
Приоритет: высокий — предпосылка для всей остальной командной функциональности.

## 1. Цель и мотивация

Сейчас платформа позволяет нескольким пользователям одновременно читать и
редактировать один vault, но не защищает их друг от друга:

- при одновременном редактировании действует «кто последний сохранил, тот и
  прав» — правки коллеги молча теряются;
- удаление заметки необратимо;
- нет истории: нельзя посмотреть, кто, когда и что изменил, и откатиться;
- пользователь не видит, что заметку прямо сейчас редактирует кто-то другой.

Цель — сделать одновременную работу команды безопасной: никакая правка не
теряется молча, любое изменение обратимо и атрибутировано.

## 2. Что уже есть (на что опираться)

| Компонент | Файл | Использование в этом плане |
| --- | --- | --- |
| Доступ к vault через интерфейс | `packages/core/ports.go` → `VaultFS` | git-слой ставится декоратором, не меняя остальной код |
| Сервис заметок | `packages/core/service.go` → `NoteService` | точки входа Save/Create/Delete |
| Событийная шина + WebSocket | `packages/core/events.go`, `packages/websocket/hub.go` | presence и уведомления об изменениях |
| JWT-клеймы с пользователем | `packages/auth/auth.go` → `Claims` | автор коммита и presence |
| Клеймы уже кладутся в контекст запроса | `packages/api/router.go` → `c.Set("user", claims)` | достать имя пользователя в хендлерах |

Важно: сейчас `NoteService` не знает, кто выполняет операцию. Первым шагом
нужно протянуть пользователя из JWT в core (см. §3.0).

## 3. Дизайн

### 3.0. Контекст пользователя в core (подготовительный шаг)

Добавить в сигнатуры мутирующих методов `NoteService` параметр
`actor string` (имя пользователя; пустая строка = аноним/авторизация
выключена):

```go
SaveNote(actor, path, content string) error
CreateNote(actor string, req CreateNoteRequest) (string, error)
DeleteNote(actor, path string) error
```

API-хендлеры берут `actor` из `c.Get("user")` (клеймы JWT). CLI передаёт
`"cli"`. События `core.Event` расширить полем `Actor string` — фронтенд
сможет показывать «изменено пользователем X» в реальном времени.

### 3.1. История изменений: git-версионирование

Идея: vault остаётся обычной папкой, но каждое изменение через платформу
фиксируется git-коммитом. Это даёт историю, diff, откат, blame и бэкап —
одним механизмом, без собственной БД (в духе ADR-0002).

Реализация:

- Новый пакет `packages/history` с интерфейсом в core:

```go
// packages/core/ports.go
type History interface {
    Record(actor, path, action string) error          // фиксирует текущее состояние файла
    Log(path string, limit int) ([]Revision, error)   // история одного файла
    Diff(path, revA, revB string) (string, error)     // унифицированный diff
    Restore(actor, path, rev string) error            // откатить файл к ревизии
    Deleted(limit int) ([]DeletedFile, error)         // «корзина»: удалённые файлы
}

type Revision struct {
    ID      string    // hash коммита
    Actor   string
    Action  string    // save | create | delete | restore
    Time    time.Time
    Message string
}
```

- Реализация — через системный `git` (exec) или библиотеку go-git.
  Рекомендация: **go-git** (github.com/go-git/go-git/v5) — не требует git в
  Docker-образе. Если возникнут проблемы производительности на больших
  vault — сменить на exec git за тем же интерфейсом.
- Репозиторий: `.git` в корне vault. При старте, если репозитория нет —
  `git init` + первичный коммит (за флагом конфига `history.enabled`).
  `.gitignore` внутри vault уважать. Каталог `.git` уже скрыт от дерева и
  индексов (фильтр скрытых файлов в `packages/filesystem`).
- Автор коммита: `actor <actor@obsidianweb.local>`.
- Сообщение: `save: Projects/Roadmap.md` / `delete: …` / `restore: … @abc123`.
- Вызовы `history.Record(...)` добавляются в `NoteService.SaveNote /
  CreateNote / DeleteNote` и в обработчик watcher-событий
  (`HandleFSEvent` → коммит с actor = `"external"`, debounce 30–60 с, чтобы
  правки из Obsidian/редактора не порождали сотни коммитов).
- Если vault уже является git-репозиторием пользователя (частый случай) —
  режим `history.mode: external`: платформа не коммитит, но читает историю.

Конфигурация (`packages/settings`):

```yaml
history:
  enabled: true        # git-история
  mode: managed        # managed | external | off
  externalDebounceSec: 45
```

API (все — permission `notes:read`, restore — `notes:edit`):

| Метод | Путь | Описание |
| --- | --- | --- |
| GET | `/api/history/{path}?limit=` | список ревизий файла |
| GET | `/api/history/{path}/diff?from=&to=` | diff двух ревизий (или ревизии и текущего состояния) |
| POST | `/api/history/{path}/restore` | `{rev}` — откат файла |
| GET | `/api/trash?limit=` | удалённые файлы |
| POST | `/api/trash/restore` | `{path}` — восстановить последнюю версию удалённого файла |

UI: на странице заметки кнопка «История» → боковая панель со списком
ревизий (автор, время), просмотр diff (перед/после), кнопка «Восстановить».
Страница «Корзина» в меню пользователя (только для `notes:edit`).

### 3.2. Защита от перезаписи: оптимистичная блокировка

- `GET /api/note/{path}` дополняется полем `contentHash` (SHA-256 содержимого).
- `PUT /api/note/{path}` принимает `{content, baseHash}`:
  - `baseHash` совпадает с хешем текущего файла → сохранить;
  - не совпадает → **409 Conflict** с телом
    `{error: "conflict", currentHash, currentContent, changedBy, changedAt}`
    (`changedBy` — из последней ревизии git, если история включена);
  - `baseHash` не передан → сохранить как раньше (обратная совместимость,
    используется CLI/интеграциями).
- Реализация в `NoteService.SaveNote`: сравнение хеша под мьютексом на путь
  (map[string]*sync.Mutex), чтобы два одновременных PUT не прошли оба.

UI при 409: диалог «Заметка изменена пользователем X, пока вы её
редактировали» с тремя действиями: показать diff (моя версия / новая
версия), перезаписать своей версией, взять новую версию (мои правки — в
буфер обмена). Полноценный трёхсторонний merge — вне scope.

### 3.3. Присутствие (presence): «кто сейчас в заметке»

- WebSocket становится двунаправленным: клиент шлёт
  `{type: "presence", path, state: "viewing" | "editing" | "left"}`
  при открытии/закрытии заметки и входе/выходе из режима редактирования.
- `packages/websocket.Hub` хранит состояние `map[path]map[username]state`
  (username — из JWT: при апгрейде соединения валидировать `?token=`,
  которое уже поддержано в middleware) и рассылает всем агрегат
  `{type: "presence.changed", path, viewers: [...], editors: [...]}`.
- Состояние очищается при разрыве соединения (readPump уже отслеживает).
- TTL-страховка: presence старше 2 минут без обновления сбрасывается.

UI: на странице заметки — аватарки просматривающих; если кто-то редактирует —
жёлтый бейдж «Редактирует: igor» и подтверждение при попытке войти в
редактор параллельно. В дереве файлов — точка-индикатор у редактируемых заметок.

### 3.4. Авторство во frontmatter

При сохранении через платформу `NoteService`:

- при создании: `created_by: <actor>` (уже генерируется `created:`);
- при каждом сохранении: `updated_by: <actor>`, `updated: <RFC3339>`.

Поля обновляются аккуратно: парсить frontmatter (`packages/shared`),
не переформатируя остальные поля пользователя (точечная замена/добавление
строк, а не полная пересериализация YAML). За флагом
`notes.trackAuthorship: true`.

## 5. Риски и решения

- **Watcher-петля**: git-коммит меняет `.git`, но watcher скрытые каталоги
  уже игнорирует; коммит содержимого файлов watcher не триггерит (файлы не
  меняются). Проверить на macOS/Linux.
- **Производительность go-git на больших vault** (10k+ файлов): коммитить
  только изменённый файл (add path, не add all); замерить на 10k файлов;
  fallback — exec git.
- **Гонки при параллельных сохранениях**: пер-путь мьютекс в NoteService
  (§3.2) охватывает и запись, и коммит.
- **Vault синхронизируется извне (Syncthing/iCloud)**: режим
  `history.mode: external` или `off`; документировать.
- **Диск**: git-история растёт; предусмотреть `obsidianweb-cli gc`
  (git gc + отчёт о размере).

## 6. Критерии приёмки

1. Два пользователя редактируют одну заметку: второй при сохранении получает
   диалог конфликта, ни одна правка не теряется молча.
2. Для любой заметки доступна история: автор, время, diff; откат к любой
   ревизии работает и сам фиксируется в истории.
3. Удалённая заметка видна в корзине и восстановима в один клик.
4. При открытии заметки видно, кто её сейчас смотрит/редактирует; индикатор
   исчезает при закрытии вкладки не позднее чем через 2 минуты.
5. Правки, сделанные напрямую в файловой системе (Obsidian), попадают в
   историю как `external` и не создают лишних коммитов чаще одного на
   debounce-интервал.
6. `go test ./...` зелёный; новые пакеты покрыты тестами (history: log/diff/
   restore/deleted; locking: конфликт и успешный путь; presence: подписка/
   отписка/TTL).
7. Документация обновлена: docs/api.md (новые эндпоинты), docs/architecture.md
   (пакет history), config.example.yaml, справка в UI (раздел про историю).
