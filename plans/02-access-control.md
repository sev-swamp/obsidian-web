# План 2. Управление доступом (когда команда растёт)

Статус: готово (реализован полностью; SSO сделан сразу с провайдером OIDC, а не только интерфейсом)
Приоритет: средний — актуален, когда пользователей больше ~5 или в vault
появляются разделы с разной чувствительностью.
Зависимости: желательно выполнить после «Плана 1» (там появляется actor в
core, который нужен и здесь), но жёсткой связи нет.

## 1. Цель и мотивация

Сейчас права глобальные: роль пользователя (viewer/editor/admin) действует
на весь vault. Для команды этого мало:

- нет разделов с ограниченным доступом («HR/ видят только двое»);
- нет зон «только чтение» для большинства («Docs/ правят только техписатели»);
- пользователи заводятся правкой config.yaml + перезапуск — недоступно
  никому, кроме администратора сервера;
- нет групп: права приходится назначать каждому персонально;
- для скриптов/интеграций приходится использовать пароль человека.

Цель: доступ по папкам, группы, управление пользователями из UI без
перезапуска, персональные API-токены и задел под SSO.

## 2. Что уже есть (на что опираться)

| Компонент | Файл | Использование |
| --- | --- | --- |
| JWT + роли + permissions в токене | `packages/auth/auth.go` | расширяется группами; permissions остаются глобальным «потолком» |
| Enforcement по permission на маршрут | `packages/api/router.go` → `requirePermission` | дополняется проверкой пути (ACL) |
| Роль → permissions, матрица | `docs/api.md` | документ расширяется ACL |
| Строгая валидация конфига | `packages/settings` (KnownFields) | новые секции валидируются так же |
| Поиск/бэклинки/дерево | `packages/search`, `packages/links`, `filesystem.Tree` | ключевое место: фильтрация результатов по правам |

## 3. Модель доступа

### 3.1. Сущности

```yaml
# users.yaml — отдельный файл (НЕ config.yaml), горячая перезагрузка
users:
  - username: igor
    passwordHash: "$2a$10$…"
    role: editor            # глобальный потолок прав (permissions в JWT)
    groups: [dev, docs]
  - username: hr-lena
    passwordHash: "$2a$10$…"
    role: editor
    groups: [hr]

groups: [dev, docs, hr]     # справочник, для валидации опечаток

acl:
  # правила проверяются сверху вниз, применяется ПЕРВОЕ совпавшее;
  # если ни одно не совпало — действует allow по глобальной роли
  - path: "HR/**"
    allow:
      - { group: hr,  access: write }
      - { user: sev,  access: write }
    default: none           # none | read | write — для всех остальных
  - path: "Docs/**"
    allow:
      - { group: docs, access: write }
    default: read
  - path: "Private/*/**"    # шаблон: Private/<username>/…
    special: owner          # доступ write только владельцу сегмента
```

Принципы:

- **Глобальная роль остаётся потолком**: ACL может только сузить права,
  но не расширить (viewer с ACL write в папке всё равно не пишет —
  у него нет permission `notes:edit` в токене). Это сохраняет понятность
  модели permissions из JWT.
- `access: none` — папка невидима (не в дереве, не в поиске, 404 на прямой
  запрос — не 403, чтобы не раскрывать существование).
- Пути — glob-шаблоны относительно корня vault (`**` — любая глубина).
- Хранение в отдельном `users.yaml`: во-первых, права правит admin через
  API, и файл перезаписывается программно (config.yaml остаётся
  «ручным»); во-вторых, его можно бэкапить/ревьюить отдельно.

### 3.2. Резолвер прав

Новый пакет `packages/acl`:

```go
type Access int // AccessNone < AccessRead < AccessWrite

type Resolver interface {
    // Access возвращает права пользователя на путь с учётом групп,
    // глобальной роли (потолок) и правил ACL.
    Access(username string, groups []string, globalMax Access, path string) Access
    // VisibleRoots — для оптимизации фильтрации дерева/поиска.
    Reload() error   // горячая перезагрузка users.yaml
}
```

- Компиляция glob-правил при загрузке (doublestar или собственный матчер).
- Кэш решений `map[user+path]Access` со сбросом при Reload.
- Если файла users.yaml нет или секция acl пуста — поведение как сейчас
  (только глобальные роли), обратная совместимость полная.

### 3.3. Enforcement (самая важная часть)

Точки, через которые содержимое может «утечь», и что в них делать:

| Точка | Файл | Действие |
| --- | --- | --- |
| GET/PUT/DELETE note, raw, attachment | `packages/api/handlers.go` | проверка `Access(path)` ≥ read/write, иначе 404/403 |
| Дерево `/api/tree` | `filesystem.Tree` → фильтр в хендлере | вырезать поддеревья с AccessNone |
| Поиск `/api/search` | результаты `SearchIndex.Search` | пост-фильтрация по Access ≥ read **до** обрезки limit |
| Бэклинки в `GetNote` | `NoteService` | фильтровать источники по Access читателя |
| Список заметок/recent | `NoteService.ListNotes/Recent` | фильтр |
| WebSocket-события | `websocket.Hub` | **пер-клиентская фильтрация**: события о файлах из скрытых папок не отправлять этому клиенту (hub должен знать username соединения — уже нужно для presence из Плана 1) |
| Создание/загрузка | `CreateNote`, `handleUpload` | Access write на целевую папку |

Архитектурно: проверки живут в API-слое и в `NoteService` (который получает
`actor` + резолвер), но НЕ в `VaultFS` — файловый слой остаётся простым.

Производительность: пост-фильтрация поиска дешёвая (проверка glob по
скомпилированным правилам ~микросекунды), но при `limit` нужно фильтровать
до усечения — передавать в `Search` предикат
`func(path string) bool` (расширение интерфейса `core.SearchIndex`).

### 3.4. Управление пользователями из UI

API (permission `settings:write`, т.е. только admin):

| Метод | Путь | Описание |
| --- | --- | --- |
| GET | `/api/admin/users` | список (без хешей) |
| POST | `/api/admin/users` | `{username, password, role, groups}` — хеш считается на сервере |
| PUT | `/api/admin/users/{name}` | смена роли/групп/пароля |
| DELETE | `/api/admin/users/{name}` | удалить (нельзя удалить себя) |
| GET/PUT | `/api/admin/acl` | чтение/запись правил ACL целиком |
| POST | `/api/admin/reload` | перечитать users.yaml (на случай ручной правки) |

Запись — через `acl.Resolver.Reload()` без перезапуска сервера. Смена
пароля/роли не отзывает уже выданные JWT (TTL 24 ч) — для мгновенного
отзыва добавить `tokenVersion` per user: число в users.yaml, копируется в
клейм при логине, middleware сравнивает; инкремент = отзыв всех токенов
пользователя.

UI: страница «Администрирование» (видна только admin, вход из меню
пользователя): таблица пользователей (создать/изменить роль и группы/
сбросить пароль/удалить), редактор ACL-правил (форма: путь-шаблон, кому,
какой доступ, default), кнопка «Проверить доступ» (введите пользователя и
путь — покажет вычисленные права; сильно упрощает отладку правил).

Совместимость: `auth.users` из config.yaml продолжают работать (читаются
как база), users.yaml имеет приоритет при совпадении имён; миграция —
командой `obsidianweb-cli migrate-users` (перенос из config.yaml в
users.yaml).

### 3.5. Персональные API-токены

Для скриптов, CI и интеграций (n8n) — токены, не зависящие от пароля:

- `POST /api/tokens {name, ttlDays?, permissions?}` → выдаёт JWT с клеймом
  `kind: "api"`, permissions ⊆ permissions пользователя (можно сузить,
  например только `notes:read`), опционально без срока;
- `GET /api/tokens` / `DELETE /api/tokens/{id}` — список и отзыв
  (отзыв — через список отозванных jti в users.yaml, проверка в middleware);
- UI: раздел «API-токены» в меню пользователя; токен показывается один раз.

### 3.6. SSO / OAuth (задел, отдельный модуль)

В спецификации проекта OAuth заявлен отдельным модулем. В рамках этого
плана — только интерфейс, чтобы модуль встал без переделок:

```go
// packages/auth
type IdentityProvider interface {
    Name() string                      // "google", "oidc", …
    AuthURL(state string) string
    Exchange(code string) (Identity, error) // → username, email, groups
}
```

Логин-страница показывает кнопки провайдеров из конфига; после Exchange
пользователь ищется/создаётся в users.yaml (`provisioning: auto | manual`),
дальше — обычный JWT платформы. Реализация конкретного провайдера
(OIDC generic — покрывает Google/Keycloak/Authentik) — отдельная задача
после ядра ACL.

## 4. Этапы и порядок работ

| Этап | Содержание | Зависимости | Статус |
| --- | --- | --- | --- |
| 1 | users.yaml + загрузчик + горячий Reload; миграция из config.yaml | — | готово |
| 2 | пакет acl: модель, glob-матчер, резолвер, тесты | 1 | готово |
| 3 | enforcement: note/raw/attachment/create/upload + tree | 2 | готово |
| 4 | фильтрация поиска (предикат в SearchIndex), backlinks, recent, WS | 3 | готово |
| 5 | admin API + страница администрирования + «проверить доступ» | 1 | готово |
| 6 | tokenVersion (отзыв сессий) + персональные API-токены | 1 | готово |
| 7 | интерфейс IdentityProvider + логин-кнопки (без провайдера) | 5 | готово (сразу с OIDC-провайдером и admin-настройкой SSO) |

Минимально полезный релиз — этапы 1–4 (доступ по папкам работает и не
протекает). Этап 5 можно вести параллельно с 3–4.

## 5. Риски и решения

- **Утечка через побочные каналы** — главный риск. Чек-лист каналов в §3.3;
  добавить интеграционный тест «пользователь без доступа к HR/ не видит
  строку из HR-заметки ни в поиске, ни в бэклинках, ни в дереве, ни в WS».
- **Усложнение модели** («у меня write в ACL, но кнопка неактивна»):
  строго держаться правила «глобальная роль — потолок» и показывать
  вычисленные права в UI («Проверить доступ»).
- **Производительность фильтрации на больших vault**: кэш резолвера,
  предикат в поиск (не строить полный результат до фильтра); бенчмарк на
  10k заметок / 50 правил.
- **Конкурентная запись users.yaml** (два админа): запись через один
  мьютекс + атомарная замена файла (tmp + rename).
- **Забытый admin-пароль при переходе на UI-управление**: config.yaml
  admin остаётся аварийным входом, читается всегда.

## 6. Критерии приёмки

1. Правило `HR/** → group hr` скрывает папку от остальных полностью:
   дерево, поиск, бэклинки, recent, прямой GET (404), WebSocket-события.
2. Правило `Docs/** default: read` даёт всем чтение, а запись — только
   группе docs; кнопки Edit/Delete у остальных скрыты (фронт получает
   эффективный доступ в ответе `GET /api/note` — поле `access: read|write`).
3. Администратор из UI создаёт пользователя, меняет группы и пароль без
   перезапуска сервера; удалить самого себя нельзя.
4. Инкремент tokenVersion мгновенно отзывает все сессии пользователя.
5. API-токен с суженными правами (`notes:read`) не может писать; отзыв
   токена действует немедленно.
6. Пустой users.yaml/отсутствие ACL → поведение идентично текущему
   (обратная совместимость подтверждена существующими тестами).
7. `go test ./...` зелёный; новые тесты: резолвер (матрица случаев),
   каждая точка enforcement, горячий reload, конкурентная запись users.yaml.
8. Документация: docs/api.md (admin API, поле access), новая
   docs/access-control.md (модель, примеры правил, FAQ), config.example.yaml.
