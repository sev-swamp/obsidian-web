# План 4. SSO как внешний плагин (сервисные токены + login-code)

Статус: проект (не начат)
Приоритет: средний — встроенный OIDC работает; план выносит его из ядра
и открывает дорогу другим провайдерам (SAML, LDAP-мост, второй OIDC).
Зависимости: план 2 (реализован: users.yaml, tokenVersion, API-токены,
встроенный OIDC).

## 1. Цель и мотивация

Сейчас SSO вшито в ядро: `packages/auth/oidc.go` (провайдер),
`packages/api/sso.go` (флоу login/callback/provisioning), конфиг в
users.yaml (`acl.SSOConfig`). Это работает, но:

- каждый новый провайдер (SAML, LDAP, кастомный корпоративный IdP) —
  правка ядра и пересборка;
- ядро тянет зависимости go-oidc/oauth2, даже если SSO выключен;
- плагинная модель проекта (`packages/plugins`, `sdk/plugin-sdk`)
  для auth не годится «в лоб»: роуты плагинов монтируются за
  `requirePermission(notes:read)` (login/callback обязаны быть
  анонимными), а выдача сессий и провижининг — trust boundary, который
  нельзя отдавать в общий Host SDK.

Цель: SSO — внешний плагин (отдельный процесс/контейнер на любом языке),
который общается с платформой по **сервисному токену** с правами на
провижининг пользователей, а сессию браузеру выдаёт платформа через
**одноразовый login-code**. Ядро перестаёт знать про OIDC.

Ключевой принцип безопасности: сервисный токен **не может выдать сессию
напрямую** — только короткоживущий одноразовый код, который браузер сам
меняет на JWT. Токен не является мастер-ключом имперсонации.

## 3. Дизайн

### 3.1. Сервисные токены

Новая сущность в users.yaml (правится только через admin API):

```yaml
serviceTokens:
  - id: sso-keycloak            # стабильный идентификатор
    name: "SSO Keycloak"
    tokenHash: "$2a$10$…"       # bcrypt от секрета; секрет показывается один раз
    permissions: [users:provision, session:code]
    roleCeiling: editor          # максимум, что токен может назначить
    revoked: false
```

- Аутентификация: заголовок `Authorization: Bearer svc_<id>_<secret>`;
  middleware находит запись по id, сверяет bcrypt, проверяет `revoked`.
- Отличие от персональных API-токенов (`/api/tokens`): не привязан к
  человеку, не JWT (отзыв мгновенный — hot-reload users.yaml уже есть),
  права задаются явно и не наследуются от роли.
- Admin API/UI: создать (секрет показывается один раз), список, отзыв.

### 3.2. Права провижининга (потолок ролей)

Новые permissions (только для сервисных токенов, в роли людей не входят):

- `users:provision` — `POST/PUT /api/service/users`: upsert пользователя
  `{username, role, groups}` без пароля (SSO-only аккаунт — вход только
  через провайдера, как сейчас в `handleSSOCallback`).
  Ограничения: назначаемая роль ≤ `roleCeiling` токена; повышать роль
  существующего пользователя выше потолка нельзя; удалять пользователей
  нельзя; менять пароли нельзя.
- `session:code` — `POST /api/service/login-code {username}` → выдаёт
  одноразовый код (см. §3.3). Только для существующих пользователей.

### 3.3. Одноразовый login-code

- `POST /api/service/login-code {username}` → `{code, expiresAt}`:
  криптослучайный код, TTL 30 секунд, одноразовый, хранится в памяти
  (map с локом; при рестарте пропадает — это нормально, пользователь
  просто логинится заново).
- `POST /api/auth/code {code}` (анонимный) → обычный сессионный JWT
  (`IssueSession` с текущим `tokenVersion` пользователя). Код сгорает
  при первом использовании и по TTL.
- Каждая выдача и обмен кода — в audit-лог (slog): токен-id, username,
  результат.
- Флоу целиком: браузер → роуты плагина → IdP → callback плагина →
  плагин по сервисному токену: upsert пользователя + login-code →
  редирект браузера на `/login?code=…` → фронтенд меняет код на JWT
  (`LoginPage` уже умеет `?sso_token=` — заменяется на обмен кода).

### 3.4. Регистрация провайдеров на логин-странице

users.yaml (правится админом или самим плагином — отдельное право
`login-providers:write` в его токене):

```yaml
loginProviders:
  - id: keycloak
    name: "Вход через Keycloak"
    url: "https://sso-plugin.example.com/start"   # роут плагина
```

- `GET /api/auth/providers` (анонимный) отдаёт список — обобщение
  текущего `/api/auth/sso/status`; LoginPage рисует кнопку на каждый
  провайдер.
- `url` ведёт на плагин; всё, что дальше (state, CSRF, редиректы IdP) —
  ответственность плагина.

### 3.5. Эталонный плагин и миграция

- Код `auth/oidc.go` + флоу из `api/sso.go` переезжают в отдельный
  сервис `plugins/sso-oidc/` (свой go-модуль, свой Dockerfile,
  compose-пример рядом с основным) — эталон для сторонних авторов.
- Порядок миграции без даунтайма:
  1. ядро получает §3.1–3.4 (аддитивно, встроенный SSO работает);
  2. плагин доводится до паритета со встроенным флоу;
  3. встроенные `/api/auth/sso/*`, `auth/oidc.go`, `SSOConfig`
     помечаются deprecated и удаляются отдельным коммитом; конфиг SSO
     из users.yaml мигрируется командой CLI в конфиг плагина.
- Docker: плагин — отдельный контейнер в общей сети; платформе о нём
  знать не нужно (связь только через loginProviders.url и REST).

Минимально полезный релиз — этапы 1–4 (инфраструктура готова, встроенный
SSO продолжает работать). Этап 6 — только после обкатки плагина.

## 5. Риски и решения

- **Компрометация сервисного токена** — главный риск. Митигируется
  дизайном: токен не выдаёт JWT (только одноразовый 30-секундный код),
  потолок ролей (нельзя произвести admin), мгновенный отзыв через
  users.yaml, audit-лог каждой операции. Рекомендация в доках: сеть
  плагин↔платформа — внутренняя (docker network), не публичная.
- **Фишинг через loginProviders.url**: url задаёт только admin (или
  токен с явным правом); UI админки показывает предупреждение при
  внешних доменах.
- **Гонка «пользователь удалён между upsert и login-code»**: login-code
  проверяет существование пользователя и tokenVersion в момент обмена
  кода, не в момент выдачи.
- **Плагин лежит** → кнопка ведёт в никуда: `/api/auth/providers` может
  опционально health-чекать url с кэшем 30 с и отдавать `available`;
  LoginPage дизейблит кнопку. (Опционально, не блокер.)
- **Обратная совместимость**: до этапа 6 встроенный SSO не трогается;
  существующие SSO-аккаунты (SSO-only, без пароля) работают с login-code
  без изменений — формат users.yaml не меняется.

## 6. Критерии приёмки

1. Плагин (эталонный sso-oidc, поднятый отдельным контейнером) проводит
   полный цикл: кнопка на логин-странице → IdP → провижининг → редирект
   с кодом → сессия в UI; ядро собрано без go-oidc/oauth2.
2. Сервисный токен с `roleCeiling: editor` не может создать admin и не
   может повысить существующего пользователя до admin (403 + audit).
3. Login-code одноразовый и сгорает за 30 с: повторный обмен и обмен
   просроченного — 401; отозванный сервисный токен перестаёт работать
   без рестарта.
4. Отзыв сессий пользователя (tokenVersion) действует и на сессии,
   выданные через login-code.
5. Пользователь без прав видит на логин-странице кнопки всех провайдеров
   из loginProviders; при пустом списке страница выглядит как сейчас.
6. `go test ./...` зелёный; новые тесты: middleware сервисных токенов,
   потолок ролей, одноразовость/TTL кода, providers API.
7. Документация: docs/api.md (service API, auth/code, providers),
   docs/sso-plugin.md (как написать свой провайдер: контракт, флоу,
   пример compose), справка в UI.
